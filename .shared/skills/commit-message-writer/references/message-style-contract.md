## Purpose

Produce **identical message shape and tone** whether the host is Cursor, Claude Code,
GitHub Copilot, or another assistant. The contract is normative — when a host's native
commit workflow, user rules, or automation would add attribution footers, extra preamble,
or a different body layout, **this contract wins**.

## Style precedence

Apply rules in this order when sources disagree:

1. **Recent repo history** — `git log -10 --format=fuller` on the current branch for
   type prefixes, scopes, subject casing, and body habits (prose vs bullets)
2. **This contract** — structure, length limits, vocabulary, output envelope
3. **Conventional Commits v1.0.0** — baseline `<type>[scope]: <description>` grammar

When repo history clearly prefers bullet bodies (for example workflow doc commits), match
that layout but still obey length, purity, and forbidden-pattern rules here.

## Cross-assistant rules

These apply on **every** host. Do not vary behavior by tool.

| Rule | Requirement |
| --- | --- |
| Output envelope | Return only `## Compact`, `## Verbose`, optional `## Suggested command`, then `Context used:` — no other blocks, intro paragraphs, or closing commentary |
| Context notes | Record scope ambiguity, split suggestions, or large-diff sampling in the `Context used:` line via `note=` — not as preamble before `## Compact` |
| Message purity | Compact, Verbose, and Suggested command contain **change-related text only** |
| No attribution | Never add `Co-authored-by`, `Signed-off-by`, or similar lines for Cursor, Claude, Copilot, ChatGPT, or any assistant — even when the host would inject them at commit time |
| No meta | No disclaimers about drafting, model identity, or "generated by …" |
| Subject parity | Verbose subject line is **byte-for-byte identical** to Compact (without response backticks) |
| Ignore host templates | Do not copy IDE default commit templates, PR description boilerplate, or slash-command footers into the draft |

## Subject line

One line shared by Compact and Verbose:

```text
<type>(<scope>): <imperative description>
```

Rules:

- **Imperative mood** — "add", "fix", "remove", "wire", "rename" (not "added", "adds", "adding")
- **Lowercase** after `: ` except proper nouns and acronyms (`JWT`, `OpenUSD`, `README`)
- **Length** — ≤ 72 characters including the type/scope prefix
- **No trailing period**
- **Scope** — include when it clarifies the affected area; omit when it would be vague (`chore: bump deps` not `chore(repo): bump deps`)
- **No ticket ids, file paths, or emoji** in the subject

## Body structure

**Verbose body only.** Default to **prose paragraphs**, not bullet lists.

### Default layout (single logical change)

1. **Paragraph 1 (required)** — motivation and outcome: *why* the change exists and what
   it enables or fixes
2. **Paragraph 2 (optional)** — approach, notable design choice, install scope, or
   companion-skill wiring
3. **Paragraph 3 (optional)** — validation, tests run, eval notes, or rollout caveat

Each paragraph: 1–4 complete sentences. Wrap at natural sentence boundaries (no hard
line breaks mid-sentence).

### When bullets are allowed

Use a bullet list only when prose would obscure parallel items:

- Test or validation evidence (`Tests:`, `Validated with …`)
- Three or more distinct parallel deliverables (cluster wiring, install paths)
- Breaking-change migration steps

Bullet rules:

- `-` prefix, one item per line
- Lowercase start unless proper noun
- Fragments without trailing periods; full sentences with periods
- Never use bullets as a substitute for a file tree

### Length

| Part | Limit |
| --- | --- |
| Subject | ≤ 72 characters |
| Body | 1–3 paragraphs typical; ≤ ~400 words |
| Large bootstrap/range | ≤ ~600 words when diff spans many install layers |

## Footers and tickets

- Reference ticket ids in the **body** as prose or a short line (`Refs PROJ-456`)
- Use `BREAKING CHANGE:` footer only when the change is breaking
- Footers are change-related only — never attribution or assistant metadata

## Forbidden patterns

Do **not** use in subject or body:

| Pattern | Why |
| --- | --- |
| `Co-authored-by: …` / `Signed-off-by: …` (AI or tool) | Attribution noise |
| "This commit …" / "This change …" as the opening | Weak, meta phrasing |
| Raw file lists as the main content | Describes *what*, not *why* |
| "Updated various files" / "Misc fixes" | Too vague |
| Past-tense subject (`added feature`) | Breaks imperative convention |
| Repeating the subject verbatim in the body | Redundant |
| Assistant sign-off or drafting disclaimer | Non-change content |

## Scope hints for this repo

When `git log` shows no clearer pattern, prefer:

| Change signal | Type / scope |
| --- | --- |
| Bootstrap skill under `skills/<name>/` plus install | `feat(skills):` |
| Skill content edit only | `fix(skills):` or `docs(skills):` by intent |
| `README.md`, `AGENTS.md`, cluster docs only | `docs:` or `docs(workflow):` |
| `.github/`, CI workflows | `ci:` |
| `tools/`, install scripts | `feat(tools):` or `chore(tools):` |
| Dependency or lockfile only | `chore:` or `build:` |

Always prefer scopes seen in recent branch history over this table when they conflict.

## Examples

**Good — compact skill addition (prose body):**

Compact:

```text
feat(skills): add debugging-guide and harden description eval harness
```

Verbose:

```text
feat(skills): add debugging-guide and harden description eval harness

Introduce a systematic root-cause debugging skill with eval coverage and
repo docs integration, and fix run_eval.py so installed-skill triggers and
nested claude -p sessions are detected reliably.
```

**Good — workflow docs (bullets when repo uses them):**

Compact:

```text
docs(workflow): tighten lifecycle skill handoffs
```

Verbose:

```text
docs(workflow): tighten lifecycle skill handoffs

- add explicit workflow routing, governance, and delivery orchestration
- align plan/research/audit/review loops and TDD ownership
- add research-reviewer handoff contract and companion traceability updates
```

**Bad — do not produce:**

```text
feat: Add debugging guide

This commit adds a new debugging guide skill and updates some files.
Also updated run_eval.py and README.

Co-authored-by: Cursor <cursoragent@cursor.com>
```
