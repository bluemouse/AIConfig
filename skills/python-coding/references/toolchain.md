# Python 3.12+ toolchain usage

Use this reference when setting up, modifying, validating, or packaging Python CLI projects.

## Interpreter and environment

Prefer invoking tools through the intended interpreter:

```bash
python --version
python -m venv .venv
# POSIX
. .venv/bin/activate
# Windows PowerShell
.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
```

Use `py -3.12` on Windows only when the Python launcher is expected. Use `python3.12` on POSIX only when multiple Python versions are installed and explicit selection is needed.

## Dependency management choices

Choose one approach and be consistent:

- **Standard library / simple**: `python -m venv`, `python -m pip`, `requirements.txt` for small scripts.
- **Modern project**: `pyproject.toml` with pinned dependencies in a lock file generated by `uv`, `pip-tools`, Poetry, Hatch, or the team-standard tool.
- **No dependency scripts**: keep external dependencies at zero when the utility must be easy to copy and run.

Use `uv` when the user wants modern fast workflow and project dependency locking:

```bash
uv init --app my-tool
uv add requests
uv add --dev pytest ruff pyright
uv run my-tool --help
```

Do not force `uv` into repositories that already standardize on `pip`, Poetry, Hatch, PDM, Conda, or Bazel.

## Recommended pyproject.toml for CLI utilities

Use this as a starting point, not a universal template:

```toml
[project]
name = "my-tool"
version = "0.1.0"
description = "Short description."
requires-python = ">=3.12"
dependencies = []

[project.scripts]
my-tool = "my_tool.cli:main_entry"

[tool.ruff]
target-version = "py312"
line-length = 100

[tool.ruff.lint]
select = ["E", "F", "I", "B", "UP", "SIM", "C4", "PIE", "RUF"]
ignore = []

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-ra"

[tool.pyright]
pythonVersion = "3.12"
typeCheckingMode = "standard"
```

For package builds, add `[build-system]` with a backend such as Hatchling, Setuptools, or Flit. Do not add packaging metadata for a single private script unless distribution or console scripts are needed.

## Formatting, linting, and imports

Use Ruff when available:

```bash
python -m ruff format .
python -m ruff check .
python -m ruff check --fix .
```

If Ruff is unavailable, use project-standard alternatives such as Black, isort, Flake8, or Pylint. Do not mix formatters with conflicting line-length or quote-style settings.

Recommended lint priorities:

1. Syntax and undefined names.
2. Import sorting and unused imports.
3. Bug-prone patterns (`flake8-bugbear` via Ruff `B`, Ruff `RUF`).
4. Modernization (`pyupgrade` via Ruff `UP`) only when it matches the baseline.
5. Style consistency after correctness.

## Static typing

Use one type checker unless the repository already uses multiple.

```bash
python -m pyright
python -m mypy src tests
```

Type-checker expectations:

- Public functions and core data transformations are annotated.
- CLI parser internals can be lightly typed if over-annotation harms clarity.
- Third-party untyped libraries may require stubs, protocols, typed wrappers, or explicit boundary casting.
- Do not silence errors broadly. Prefer narrow `cast()`, wrapper functions, or local `# type: ignore[code]` with a reason.

## Testing tools

Use `pytest` when third-party dev dependencies are allowed:

```bash
python -m pytest
python -m pytest -q
python -m pytest --cov=src --cov-report=term-missing
```

Use `unittest` when standard-library-only is required:

```bash
python -m unittest discover -s tests
```

Use `tox`, `nox`, or CI matrixes when testing multiple Python versions or optional dependency sets.

## Syntax and import validation

Always run at least one fast correctness gate:

```bash
python -m compileall -q src tests
python -m py_compile path/to/script.py
python -m your_package --help
```

For a single-file script:

```bash
python -m py_compile script.py
python script.py --help
```

## Debugging tools

- `breakpoint()` for local debugging; respects `PYTHONBREAKPOINT`.
- `python -m pdb script.py args...` for standard-library interactive debugging.
- `faulthandler` for low-level crashes or hangs.
- `logging` with DEBUG level for traceable CLI behavior.
- `traceback` module at the top-level boundary only when developer diagnostics are requested.

## Profiling and performance tools

Start with measurement:

```bash
python -m timeit 'sum(range(1000))'
python -m cProfile -o profile.out script.py args...
python -m pstats profile.out
```

Use third-party profilers such as `py-spy`, `scalene`, or `memray` only when allowed and useful. Optimize algorithms and IO first; micro-optimizations come last.

## Packaging and distribution

For installable CLIs, use console script entry points in `pyproject.toml`. Avoid relying on executable bits or shebang behavior alone when distributing cross-platform.

Common commands:

```bash
python -m build
python -m twine check dist/*
python -m pip install -e .
python -m pipx install .
```

Use `pipx` for user-level CLI installation when packaging a command-line app.

## CI quality gate template

A minimal CI or local pre-submit gate should run:

```bash
python -m compileall -q src tests
python -m ruff format --check .
python -m ruff check .
python -m pyright
python -m pytest -q
```

Adapt paths and tools to the repository. Never present a gate as completed unless it was actually run.
