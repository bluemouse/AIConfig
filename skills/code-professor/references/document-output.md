# Document output

By default, return the guide in the conversation. Write files only when the user explicitly requests saved documentation.

## When to write files

Write Markdown files when the user asks to:

- save, write, or export the guide
- create documentation under a specific path
- add a doc to the repository for onboarding or reference

Do not write files proactively "to be helpful" unless the user asked.

## Path conventions

1. Use the path the user specifies.
2. If no path is given, suggest `docs/code-professor/<deliverable-slug>.md` (e.g. `docs/code-professor/repository-orientation.md`, `docs/code-professor/caching-module-deep-dive.md`).
3. Confirm before overwriting an existing file.
4. Do not commit unless the user explicitly asks.

## File header

Include at the top of written files:

```markdown
<!-- Generated by code-professor. Scope: [brief scope]. Depth: [overview|standard|deep]. -->
```

Add generation date when useful. Keep the header minimal.

## Content rules

- Written files use the same evidence and citation standards as in-conversation output.
- Refresh all `path:line` references after any temporary instrumentation cleanup.
- Do not include transient experiment logs or snapshot paths in permanent docs unless the user wants them.

## What not to do

- Do not create Jira tickets, Confluence pages, or wiki entries unless explicitly asked (use host tooling when available).
- Do not replace existing project documentation without confirmation.
- Do not write generated output into build artifact directories.
